# Сервис очистки персональных данных (PII-фильтр)

Прослойка для корпоративного LLM-гейтвея: принимает текст или файл, возвращает текст с обезличенными персональными данными. Фронтенда нет — один интерфейс `/clean` и для ручной, и для автоматической подачи (по постановке). Маршрутизация/прокси — вне зоны сервиса.

## Архитектура: каскад, дёшево → умно

```
текст / файл (PDF, DOCX, TXT, MD, HTML -> текст обычными библиотеками;
              .doc и PDF-сканы с битым OCR -> явный ОТКАЗ, см. находки 2-3)
  └─ Слой 1: regex          — жёсткие форматы: телефоны (мобильные, городские с 3-5-значным
                              кодом, с префиксом +7/8 и без), email, СНИЛС, ИНН/КПП/ОГРН,
                              паспорта, карты, р/с, ДР с маркером, адреса. ~0.1-30 мс/док.
  └─ Слой 2: NER (Natasha)  — ФИО (вкл. «Фамилия И.О.»), организации и филиалы.
                              С фильтром шума: аббревиатуры (ЕИС, НМЦ), слова-роли
                              («Заказчик»), ссылки на кодексы НЕ трогаются. ~0.4-1.6 с/док.
  └─ Эвристика года         — даты с годом вне 1930-2012 не могут быть датой рождения ->
                              оставляются автоматически, без LLM.
  └─ Слой 3: LLM (Ollama,   — ТОЛЬКО оставшиеся спорные даты («родился 12.05.1985» vs
     опционально, PII_LLM=on)  «согласовано 12.05.1985»). Модель локальная — данные не
                              покидают контур (в этом смысл задачи). Safe-режим: при
                              недоступности/ошибке LLM сомнительное вырезается.
```

Замена — стабильные плейсхолдеры: один и тот же человек во всём документе → всегда `{PERSON_1}` (текст остаётся связным для LLM на принимающей стороне). Сервис возвращает словарь замен → возможна обратная подстановка в ответ модели (де-анонимизация).

## API

```
POST /clean        form-data: text=... ИЛИ file=<документ>
POST /clean/json   {"text": "..."}
GET  /health       -> {"status": "ok", "llm_layer": true|false}
```

Ответ: `cleaned_text`, `replacements` (плейсхолдер→оригинал), `entities_found`, `timings` (мс по слоям), `llm_usage` (фактические токены). При повреждённом входе — `error` вместо результата.

## Метрики

### Синтетический корпус (80 документов, 406 ПД-сущностей, 181 keep-сущность)

| Метрика | Значение |
|---|---|
| Recall (вычищено ПД) | 100% по всем 15 типам |
| Keep-rate (не тронуто лишнего) | 100% |
| Задержка (regex+NER) | медиана 3-4 мс/док, p95 ~6 мс |

Оговорка: 100/100 — верхняя граница (корпус и детекторы согласованы по форматам); реальная точность калибровалась на живых документах (ниже).

### LLM-слой (замеры на живом сервисе, qwen2.5:3b-instruct / Ollama, CPU)

| Метрика | Значение |
|---|---|
| Задержка | ~5.2 с (тёплая модель) / ~18 с (холодный старт) на запрос |
| Токены | ~370 in / ~32 out на запрос с 1 кандидатом |
| Когда вызывается | только при наличии немаркированных дат с «годом рождения» |

## Проверка на реальных документах (zakupki.gov.ru)

Прогнано на 5 реальных файлах закупок: документация АО «Почта России» (87 тыс. символов), ТЗ, договор и извещение МДОАУ (сканы), .doc-файл. В очищенных версиях реальных .docx: 0 утечек ФИО/телефонов/email, ссылки на 44-ФЗ/223-ФЗ и даты договоров сохранены. Найдено и исправлено **7 классов проблем**:

1. **Кодировка консоли Windows** (cp1251) ломала вывод демо — принудительный UTF-8.
2. **Старый .doc (Word 97-2003)** давал бинарный мусор «на очистку» — теперь явный отказ по расширению и бинарной сигнатуре файла.
3. **PDF-сканы с некачественным OCR-слоем: телефон и ФИО реально утекали при формально «успешной» очистке** («Ten+1(3463)...» не ловится паттернами, «Merqepqrcosa» не распознаётся NER). Добавлен детектор повреждённого текста (доля кириллицы < 50% → отказ). Принцип: молчаливая деградация опаснее отказа — гейтвей получает error, а не «очищенный» текст с утечкой.
4. **Городские телефоны с 4-значным кодом** (`+7 (3463) 45-50-17`) не ловились «мобильным» паттерном 3-3-2-2 — переписан на «+7/8 + ровно 10 цифр с разделителями», с lookaround-защитой от срабатываний внутри счетов и реестровых номеров.
5. **Городские телефоны без префикса +7/8** (`Факс (423) 241-21-43`) утекали — добавлены контекстный (тел/факс) и форматный паттерны.
6. **NER-шум на юридических текстах**: аббревиатуры (ЕИС, ЭП, НМЦ, ТРУ), слова-роли («Заказчик», «Общество», «Комиссия») и правовые ссылки («Гражданского кодекса», «Правительства РФ») помечались как ORG — документ терял смысл (45 сущностей, ~половина мусор). Добавлен фильтр-стоплист; после — 27 сущностей, юридический контекст цел.
7. **LLM-слой перестраховывался**: qwen2.5:3b вырезала даты договоров вместе с потенциальными датами рождения (KEEP-инструкции игнорировались — тот же паттерн, что в задаче 2 до тюнинга). Решение — детерминированная эвристика до LLM: даты с годом вне 1930-2012 не могут быть ДР и оставляются автоматически; LLM получает только правдоподобных кандидатов. Одновременно чинит keep-rate и снижает токены/латентность в разы.

Общий вывод (подтверждён дважды — здесь и в задаче 2): маленькой локальной модели нельзя отдавать решения, которые можно принять детерминированно; LLM — только для неустранимой неопределённости.

## Зафиксированные решения по политике (обсуждаемо с заказчиком)

- **Госорганы, кодексы, правовые ссылки** — не ПД (публичный правовой контекст), не обезличиваются. Меняется одной строкой стоп-листа в `ner_layer.py`.
- **к/с банка** — публичный реквизит банка (общий для всех клиентов), не обезличивается; р/с организации — обезличивается.
- **Города вне адресного контекста** («поставка в г. Москва») — не ПД; адреса с маркером («зарегистрирован по адресу: ...») — ПД.

## Известные ограничения

- .doc и сканы без текстового слоя отклоняются (нужно пересохранение/OCR, напр. Tesseract) — осознанное решение, не дыра.
- Один субъект в разных падежах/обрамлениях может получить разные плейсхолдеры (морфологическая нормализация — следующий шаг).
- Модель 3B на CPU: LLM-слой секунды на запрос; альтернатива для прода — лёгкий классификатор (ruBERT-tiny2) или модель крупнее.

## Про Microsoft Presidio (упомянут заказчиком)

Presidio — фреймворк ровно этого назначения (NER + регексы + анонимизация), но англо-центричный: для русского нужны кастомная NLP-модель и recognizers под российские форматы (СНИЛС, ИНН, паспорт РФ) — большая часть работы всё равно своя. Наш каскад — «Presidio, собранный под русский» + LLM-слой для контекста, которого в Presidio нет из коробки. Следующий шаг — прогнать Presidio с ru-моделью на том же корпусе для честного сравнения.

## Запуск

```bash
pip install -r requirements.txt
python generate_corpus.py            # синтетический корпус с ground truth
python evaluate.py                   # метрики (regex + NER)
python demo.py [файл]                # демо «до/после»
uvicorn app.main:app --port 8010     # сервис (LLM-слой: PII_LLM=on + запущенная Ollama)
# или: docker build -t pii-filter . && docker run -p 8010:8010 pii-filter
```

Реальные документы для тестов брались с zakupki.gov.ru (открытые данные); в репозиторий не включены намеренно — они содержат ФИО и контакты реальных людей (`data/real/` в .gitignore).
